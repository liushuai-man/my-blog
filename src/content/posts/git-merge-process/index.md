在多人协作的项目开发中，代码合并是贯穿始终的核心操作——它不仅关系到个人开发成果的落地，更直接影响团队代码的一致性、项目的稳定性。尤其是将个人开发分支的代码，规范合并到团队共用的develop分支，每一步都需要严谨操作，避免出现代码冲突、功能异常等问题。

结合日常项目实战经验，今天就详细拆解代码合并的完整步骤，重点说明从个人分支开发完成，到最终推送合并代码至develop分支的全过程，适合刚接触团队协作开发、对代码合并流程不熟悉的小伙伴参考，也可作为日常操作的备忘录。

### 一、前置准备：明确分支规范（基础前提）

在开始代码合并前，必须先明确团队的分支管理规范——这是所有操作的基础，避免因分支命名混乱、操作流程不统一，导致合并失败或代码污染。

常规项目的分支规范（Git为例），通常分为三类核心分支：

*   master/main 分支：生产环境分支，存放正式上线的代码，严禁直接在该分支开发或合并代码；
    
*   develop 分支：开发环境分支，团队共用的开发分支，所有个人开发完成的功能，最终都需要合并到该分支，用于后续测试、联调；
    
*   个人开发分支（feature分支）：基于develop分支创建，命名通常遵循「feature-功能名称-开发者」或「feature-需求ID」格式（如 feature-user-login-zhangsan），个人所有开发工作都在该分支进行，不直接操作develop分支。
    

本文的核心的是「个人feature分支 → develop分支」的合并流程，所有操作均基于Git命令行（也可对应Git客户端，如SourceTree、GitKraken的可视化操作，逻辑一致）。

### 二、代码合并核心步骤（从开发完成到提交合并）

整个流程分为「个人分支自检」「拉取远程develop最新代码」「解决冲突」「提交合并请求」「审核合并」「推送确认」6个关键步骤，一步都不能少，每一步都有需要注意的细节。

#### 步骤1：个人分支开发完成，本地自检（避免无效合并）

当个人负责的功能在feature分支开发完成后，不要急于合并，先在本地完成自检，确保代码本身无问题，减少合并后出现异常的概率。

自检核心操作：

1.  切换到个人feature分支（确保当前操作分支正确），命令：`git checkout feature-xxx`（xxx为自己的分支名称）；
    
2.  查看本地代码修改记录，确认修改内容无误，没有多余的临时代码、注释，命令：`git status`（查看未提交的修改）、`git log`（查看提交记录）；
    
3.  本地运行项目，执行单元测试、功能测试，确认开发的功能正常，无语法错误、运行异常，这一步是关键——避免将有bug的代码合并到develop分支，增加团队测试成本；
    
4.  提交本地未提交的修改，确保本地代码与个人分支的远程代码同步，命令：`git add .`（添加所有修改）、`git commit -m "feat: 完成xxx功能开发（详细描述修改内容）"`（提交注释需规范，便于后续追溯）。
    

注意：提交注释建议遵循规范（如Angular规范），区分feat（新功能）、fix（修复bug）、docs（文档修改）等，避免模糊的注释（如“修改代码”“提交”）。

#### 步骤2：拉取远程develop分支最新代码（避免冲突前置）

多人协作中，develop分支会不断有其他同事合并代码，若直接将自己的feature分支合并到develop分支，很可能因为代码版本不一致，出现严重的代码冲突。因此，在合并前，必须先将远程develop分支的最新代码拉取到本地，与自己的feature分支合并，提前解决冲突。

操作步骤：

1.  本地切换到develop分支，命令：`git checkout develop`；
    
2.  拉取远程develop分支的最新代码（同步团队最新修改），命令：`git pull origin develop`；
    
3.  拉取完成后，再次切换回个人feature分支，命令：`git checkout feature-xxx`；
    
4.  将本地develop分支的最新代码，合并到当前feature分支，命令：`git merge develop`。
    

这一步的核心目的：将团队最新的代码同步到自己的开发分支，提前发现并解决冲突，避免后续合并到远程develop分支时，出现冲突难以处理的情况（尤其是多人修改同一文件、同一代码块时）。

#### 步骤3：解决代码冲突（合并的核心难点）

执行`git merge develop`后，若本地feature分支与develop分支有代码冲突（如同一行代码被不同人修改、同一文件被删除/修改等），Git会提示“conflict”，并标记出冲突文件。此时，必须先彻底解决所有冲突，才能继续后续操作。

冲突解决步骤与注意事项：

1.  查看冲突文件：Git会在冲突文件中，用「<<<<<<< HEAD」「=======」「>>>>>>> develop」标记出冲突部分，其中「HEAD」后面的是自己feature分支的代码，「develop」后面的是远程develop分支的最新代码；
    
2.  解决冲突：打开冲突文件，结合业务逻辑，判断保留哪部分代码、修改哪部分代码——若自己的修改与团队代码不冲突，可保留双方代码；若冲突（如逻辑不一致），需与相关开发同事沟通，确认最终代码（严禁擅自删除或修改他人的核心代码）；
    
3.  冲突解决后，删除Git标记的冲突符号（<<<<<<<、=======、>>>>>>>），保存文件；
    
4.  提交冲突解决后的代码，命令：`git add .`、`git commit -m "fix: 解决与develop分支的代码冲突（说明冲突内容，如xxx文件的xxx代码块）"`；
    
5.  再次本地运行项目，测试功能是否正常——避免解决冲突时，误删或修改代码，导致功能异常。
    

提示：若冲突较多、较复杂，可借助Git客户端的可视化工具（如SourceTree）解决，更直观，不易出错。

#### 步骤4：推送个人feature分支到远程仓库

解决完冲突、确认本地代码无问题后，将自己本地feature分支的最新代码（包含自己的开发代码+合并的develop最新代码+冲突解决后的代码），推送至远程仓库的对应feature分支，便于后续提交合并请求。

操作命令：`git push origin feature-xxx`（xxx为自己的分支名称）。

注意：若首次推送该feature分支，Git可能会提示关联远程分支，按照提示执行命令即可（如`git push --set-upstream origin feature-xxx`）；若推送失败，大概率是远程feature分支有其他修改，需先拉取远程feature分支代码（`git pull origin feature-xxx`），解决冲突后再推送。

#### 步骤5：提交合并请求（MR/PR），等待团队审核

远程feature分支推送完成后，需要提交「合并请求」（Merge Request，简称MR；GitHub中称为Pull Request，简称PR），请求将远程feature分支的代码合并到远程develop分支。这一步是团队协作的关键，用于确保合并的代码符合团队规范、无bug。

操作步骤（聚焦GitLab平台，实操性拉满，适配日常团队协作场景）：

1.  登录GitLab/GitHub，进入当前项目仓库；
    
2.  找到「合并请求」入口，点击「新建合并请求」；
    
3.  选择「源分支」和「目标分支」：源分支选择自己的远程feature分支（feature-xxx），目标分支选择远程develop分支（必须确认无误，严禁选错分支，尤其是不能选master/main分支）；
    
4.    提交MR：确认所有信息填写无误后，点击GitLab页面底部“创建合并请求”，完成MR提交，此时页面会跳转至MR详情页，可实时查看审核状态。
    
    1.  标题：遵循GitLab团队规范，清晰标注合并类型+内容（如“【feat】合并feature-user-login分支到develop，完成用户登录功能”），避免模糊表述；
        
    2.  描述：详细填写3类核心信息——① 开发背景（关联的需求ID、开发目的）；② 修改内容（涉及的文件、核心代码变更）；③ 自检说明（明确标注“已拉取develop最新代码、已解决所有冲突、本地功能测试正常”），若有需求文档、测试报告，可通过GitLab“附件”功能上传，或粘贴链接；
        
    3.  指定审核人：在GitLab“审核人”下拉框中，选择团队负责人、代码OWNER（对应模块负责人），建议至少指定1名审核人，确保代码质量可控；
        
    4.  补充设置（可选但推荐）：勾选“合并前执行CI/CD流水线”（GitLab自带功能，自动检测代码语法、执行单元测试，避免不合格代码合并）；取消“允许强制推送”，防止误操作污染分支；若功能未完全开发完毕，可勾选“草稿”（Draft），标注“待完善，暂不审核”，避免误合并。
        
5.  填写合并请求（MR）核心信息，确保审核人能快速了解合并内容，提升审核效率：
    

#### 步骤6：MR审核与修改（GitLab核心审核流程）

MR提交后，需等待审核人审核，这是GitLab协作中保障代码质量的关键环节，需配合审核人完成修改，直至审核通过。

核心操作与注意事项：

1.  查看审核状态：在GitLab个人中心“合并请求”列表，或直接进入该MR详情页，可查看审核人是否“通过”“驳回”，或有“修改建议”；
    
2.  处理审核意见：若审核人提出修改建议（如代码规范不达标、逻辑有漏洞），需返回本地feature分支，按照建议修改代码，修改完成后，重复“提交本地修改→推送远程feature分支”操作（无需重新创建MR，GitLab会自动同步最新代码到该MR）；
    
3.  重新请求审核：代码修改完成并推送后，在GitLab MR详情页，点击“提醒审核人”，告知审核人重新审核，直至审核通过；
    
4.  特殊情况处理：若审核过程中，有其他同事提交MR并合并到develop分支，导致当前MR出现“与develop分支冲突”提示，需重新执行“拉取远程develop最新代码→解决冲突→推送远程feature分支”，同步更新MR。
    

#### 步骤7：合并MR（GitLab操作，完成最终合并）

当MR审核通过、CI/CD流水线执行通过（若有）、无代码冲突时，即可在GitLab上完成最终合并，将远程feature分支的代码合并到远程develop分支。

GitLab合并操作步骤：

1.  进入该MR详情页，确认页面顶部显示“可合并”（Mergeable）状态；
    
2.  点击页面右侧“合并”按钮（Merge），选择合并方式（GitLab默认提供3种，推荐团队统一规范：① 常规合并（Merge commit）：保留所有提交记录，便于后续追溯；② 压缩合并（Squash and merge）：将feature分支的所有提交压缩为1条，保持develop分支提交记录简洁；③ 快进合并（Fast-forward merge）：仅当feature分支基于最新develop分支创建、无额外冲突时可用）；
    
3.  合并完成后，GitLab会提示“合并成功”，同时可选择“删除源分支”（即远程feature分支）——推荐删除，避免远程分支过多，造成分支混乱（本地feature分支可保留，用于后续bug修复或功能迭代）。
    

#### 步骤8：验证合并结果（收尾关键一步）

MR合并完成后，并非万事大吉，需在GitLab上验证合并结果，确保代码成功同步到develop分支，避免出现合并遗漏。

GitLab验证操作：

1.  进入项目仓库，点击左侧“分支”，找到develop分支，点击进入；
    
2.  查看develop分支的“提交记录”，确认自己feature分支的提交记录已同步（可通过提交注释、修改文件核对）；
    
3.  （可选）若有测试环境自动部署功能（GitLab CI/CD配置），可等待测试环境部署完成后，访问测试环境，验证合并的功能是否正常，确保无异常。
    

### 三、GitLab代码合并核心注意事项（避坑指南）

结合GitLab实操经验，整理几个高频避坑点，新手必看，避免因操作不当导致代码问题：

*   严禁直接修改远程develop分支：GitLab中可通过分支保护设置（项目→设置→仓库→分支保护），禁止普通开发者直接推送代码到develop分支，所有合并必须通过MR完成；
    
*   MR提交前务必自检：确保本地已解决所有冲突、功能测试正常，避免提交“半成品”MR，浪费审核人时间；
    
*   重视GitLab CI/CD流水线：若项目配置了流水线，MR合并前需确保流水线执行通过，若流水线失败（如语法错误、测试不通过），需先排查问题，再提交审核；
    
*   合并后及时同步develop分支：MR合并完成后，建议本地切换到develop分支，执行`git pull origin develop`，同步最新代码，便于后续开发新功能时，基于最新版本创建feature分支；
    
*   MR描述需详细：避免简单填写“提交代码”“修改bug”，详细的描述能帮助审核人快速了解修改内容，也便于后续出现问题时，快速追溯原因。
    

### 四、总结

以上就是基于GitLab平台，从个人feature分支开发完成，到最终合并代码至develop分支的完整流程——核心逻辑是“本地自检→同步团队代码→解决冲突→GitLab提交MR→审核修改→合并验证”，每一步都需严谨操作，既要保障个人代码的正确性，也要兼顾团队协作的规范性。

其实代码合并的核心不是“执行命令”，而是“规范协作”，尤其是在多人协作项目中，遵循GitLab分支规范、MR审核规范，才能减少代码冲突、提升开发效率，避免因合并操作不当导致项目异常。
